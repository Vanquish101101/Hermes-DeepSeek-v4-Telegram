# Инструкция к оркестрации 1 (Foresight)

> Подход «Foresight»: **лёгкая оркестрация** — Hermes решает задачи сам, опираясь на
> правильный выбор MCP-инструментов (поиск/исследование/скрап/память), без тяжёлого
> департамента субагентов. Альтернатива — «Инструкция к оркестрации 2 (Мой вариант)».

Версия: **1 (Foresight)**. Дата старта: 21.06.2026.
Связано: `SOUL.md` Hermes (правила), `Departments/Инструкция к оркестрации 2 (Мой вариант).md` (альтернативный алгоритм, на паузе).

---

## Идея подхода
Вместо департамента из 31 субагента — Hermes как один сильный исполнитель, который для каждой
подзадачи берёт оптимальный инструмент по типу и стоимости (приоритет MCP). Дёшево, быстро,
прозрачно. Разведка и факты — через Perplexity (ответ из живого веба с источниками).

### Этап 1 (пункт «а») — Роутинг инструментов поиска/исследования в SOUL.md

| Задача | Инструмент |

**Ниша Perplexity:** «вопрос → синтезированный ответ из живого веба **с источниками**». Модели по стоимости: `sonar` → `sonar-pro` → `sonar-reasoning-pro` → `sonar-deep-research` (брать минимально достаточную; для `sonar` `max_tokens` ≥ 16).

### Этап 2 (пункт «б») — Perplexity как основной инструмент OSINT/разведки
В шаге разведки (когда он применяется) основной инструмент — Perplexity MCP (`sonar`/`sonar-pro` для фактов с источниками, `sonar-deep-research` для отчёта по нише/конкурентам); дополняют Firecrawl (конкретные URL), Apify (скрап), x_search (соцсети). Использовать как вспомогательную информацию и образцы — **НЕ копирование**.

### Этап 3 — Подключение и роль Firecrawl (21.06.2026)
Firecrawl подтверждён эмпирически (прямой запрос к `api.firecrawl.dev`, ключ `fc-8105…` валиден) и встроен
в Правило 5 `SOUL.md` как **«парсер конкретного URL/сайта»** — пара Perplexity по разведке: Perplexity отвечает
на вопрос «что происходит в нише», Firecrawl — «дай мне реальный контент вот этого сайта/страницы».

Проверены 4 режима (см. `Hermes-config/Firecrawl-setup.md` для деталей и примеров):
- **`scrape`** — страница → чистый markdown (1 кредит)
- **`map`** — сайт → список всех его URL (0 кредитов, бесплатно)
- **`search`** — лёгкий веб-поиск без синтеза, просто ссылки+сниппеты (≈1.3 кредита/результат)
- **`v2/scrape` с форматом `json`** — страница → структурированные данные по схеме (5 кредитов); заменяет
  устаревший `/v1/extract` (тот же результат, но 22 кредита — **не использовать**)

В пайплайне разведки конкурентов (Foresight / `market-competitor-analyst`) роль Firecrawl: `map` сайта
конкурента → выбрать релевантные страницы (цены/фичи/блог) → `v2/scrape json` по каждой с нужной схемой
(цена, тарифы, фичи) → готовая таблица для сравнения. Дешевле и точнее, чем просить LLM пересказать сайт
по памяти.

### Этап 4 — OpenRouter (расширенный роутинг моделей) (21.06.2026)
OpenRouter подтверждён эмпирически (`GET /api/v1/auth/key` → ключ валиден; реальный `chat_completion` на
`anthropic/claude-haiku-4.5` → HTTP 200, `usage.cost = $0.000034`) и переписан в Правило 4 `SOUL.md`
с нуля — обнаружен и исправлен реальный баг (id моделей были с дефисом вместо точки, что вернуло бы
404/400 при первом реальном вызове).

Два независимых механизма, не сводящихся к одному вызову модели:
- **`chat_completion` (MCP)** — разовый вызов другой модели под конкретную подзадачу (vision, гигантский
  контекст, формальная логика, дешёвый быстрый ответ) — таблица в `SOUL.md` §4.1.
- **Pareto Code Router** (`openrouter/pareto-code` + `min_coding_score: 0.65`) — авто-выбор самой дешёвой
  модели, проходящей порог качества кода. Подтверждено в исходнике, что порог model-gated — работает
  только на этой конкретной модели.
- **Response cache** (`response_cache`/`response_cache_ttl`) — повтор идентичного запроса в течение 300с
  бесплатен (заголовки `X-OpenRouter-Cache`).
- **Fallback-провайдер** — поддерживается (~30 провайдеров), но `fallback_providers: []` пока пуст,
  осознанно не активирован (поведенческое изменение для прода).

DeepSeek v4 (`deepseek-chat`) остаётся основной моделью — OpenRouter не заменяет её, а дополняет точечными
вызовами. Детали и примеры — `Hermes-config/OpenRouter-setup.md`.

**Добавление (22.06.2026, версия 1.5.0) — MiniMax M3:** найдена и подключена модель `minimax/minimax-m3`
через тот же канал OpenRouter (без новых секретов). Контекст 1 048 576 токенов, цена $0.30/$1.20 за 1M —
дешевле всех моделей в таблице роутинга. Проверена реальной задачей (логическая загадка + код на Python —
оба решения верны), не просто пингом. Найдена и задокументирована особенность: один из backend-провайдеров
OpenRouter (`AtlasCloud`) в 1 из 4 тестов продублировал ответ целиком; решение (`provider_routing.ignore`)
готово, но не активировано — глобальная настройка, трогаем только по явному запросу. Позиционирована в
`SOUL.md` §4.1/§4.5/§9 как модель, которую пробуем ПЕРВОЙ для длинных документов и reasoning/код-задач.

### Этап 5 — Notion как база знаний находок (22.06.2026, версия 1.6.0)
Notion был подключён как MCP ещё в версии 0.0.5 (15.06.2026), но до этого этапа не имел роли в
оркестрации — ключ работал, но ни одного правила «когда использовать» не было. Теперь встроен как
**третий, отдельный слой** рядом с роутингом инструментов (§5) и учётом расходов (§8):

- **Архитектура (эмпирически создана 22.06.2026):** корневая страница `Foresight — База знаний (Hermes)`
  → база данных `Исследования и находки` (свойства `Name`/`Дата`/`Инструмент`/`Категория`/`Стоимость ($)`/
  `Версия проекта`). Не заменяет `cost-ledger.md` (финансы) и не заменяет `SOUL.md` (правила) — это архив
  **содержательных находок**, не каждого мелкого запроса.
- **Связь с уже подключённым пайплайном:** после значимого результата от Perplexity/Firecrawl/OpenRouter
  (рыночные данные, особенность модели/провайдера) — сохранять находку в Notion. Перед новым платным
  исследованием — сначала `search` в этой базе (правило экономии §9): дешёвый Notion-поиск вместо
  повторного платного запроса.
- **Первая запись в базе** — находка AtlasCloud/MiniMax M3 (см. Этап 4 выше, §4.5 `SOUL.md`) — перенесена
  из текста SOUL.md в Notion как пример «как выглядит находка в базе».
- Найден и исправлен баг кодировки: создание страниц с кириллицей через `curl`+heredoc давало битый
  UTF-8 — исправлено переходом на Python `urllib.request` с явным `ensure_ascii=False`. Корректность
  проверена побайтово (hex заголовка), а не на глаз — терминал искажал кириллицу чисто визуально.
- Подробности и проверка — `Hermes-config/Notion-setup.md`; правило — `SOUL.md` §10 (+ строка в §5).

### Этап 8 — Vercel как слой статуса деплоя (22.06.2026, версия 1.9.0)
Vercel подключён как community MCP (`@open-mcp/vercel`) ранее, но не имел роли в оркестрации. Встроен
как **слой статуса деплоя/наблюдаемости** — в отличие от PostMyPost (выход/публикация с подтверждением)
Vercel в первую очередь читает состояние и докладывает, запись — отдельная, осторожная операция:

- **Проверка эмпирически:** прямые REST-вызовы `https://api.vercel.com` (`GET /v2/user`, `/v9/projects`,

- **⚠️ Жёсткое ограничение:** никаких изменений прод-окружения без явной команды пользователя — push
  в GitHub не деплоит автоматически, новый деплой по умолчанию preview, alias главного домена менять
  только по прямой команде. Правило перенесено из отдельного проекта лендинга CORE и закреплено как

- **Конкретная связь сервисов:** если в логах деплоя находится значимая техническая находка — заносить

- Подробности и проверка — `Hermes-config/Vercel-setup.md`; правило — `SOUL.md` §13.

## Статус
- Подход **рабочий**: его ядро (Правило 5) уже живёт в `SOUL.md` и применяется Hermes.
- Инструменты, подключённые и встроенные в роутинг: **Perplexity** (вопрос→ответ с источниками),

- Дальнейшее расширение — по тому же принципу: добавлять категории в матрицу инструментов
