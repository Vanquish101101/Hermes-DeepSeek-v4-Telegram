## 1. GitHub и версионирование проекта

При любом значимом изменении проекта (подключение нового агента, MCP-сервера, ресурса, изменение конфигурации):

- Спросить пользователя про соответствующий GitHub-репозиторий проекта

- Каждое сохранение оформлять по формату:
  ```
  Название | краткое описание изменения | version X.X.X
  ```
  Например: `Hermes Config | GitHub MCP подключён | version 0.0.2`
- Вести файл `CHANGELOG.md` (или `VERSION.md`) в папке проекта с описанием сделанных изменений и текущей версии

В будущем это будет автоматизировано. Пока — предлагать явно.

## 2. MCP — приоритет над API

Перед подключением любого внешнего ресурса или сервиса:

1. Сообщить пользователю: _"Ищу способ подключения через MCP, подождите..."_
2. Изучить сайт и документацию подключаемого ресурса на предмет MCP-поддержки
3. Если MCP доступен — предложить и использовать **только MCP**, не API
4. Если MCP недоступен — только тогда переходить к API-подключению
5. Объяснить разницу: MCP стабильнее, глубже интегрируется, не требует ручного управления ключами в коде

## 4. Роутинг моделей через OpenRouter

OpenRouter подключён в Hermes ДВУМЯ способами одновременно:

1. **MCP-инструмент** `chat_completion` из `@mcpservers/openrouterai` — для разового вызова другой модели прямо в разговоре.
2. **Нативный провайдер** Hermes (`openrouter:` в `config.yaml`) — отвечает за кэш ответов и роутер кода (ниже). Основная модель остаётся `deepseek-chat` (провайдер `deepseek`) — OpenRouter не заменяет DeepSeek v4, а дополняет его точечными вызовами.

### 4.1 Когда вызывать другую модель через `chat_completion`

| Ситуация | Модель (ТОЧНЫЙ id OpenRouter, через точку, не дефис!) |
|----------|--------|
| Задача содержит изображение или скриншот | `anthropic/claude-sonnet-4.6` (vision) |
| Очень длинный документ (>80 000 символов) | `anthropic/claude-opus-4` (200k контекст) |
| Сложная математика или формальная логика | `openai/o3-mini` |
| Нужен быстрый дешёвый ответ | `anthropic/claude-haiku-4.5` |
| Нужен качественный код по минимальной цене | `openrouter/pareto-code` (см. 4.2) |
| Всё остальное | Отвечай сам (DeepSeek v4) |

Не используй OpenRouter для простых задач — только когда есть явное преимущество другой модели.

> ⚠️ Найден и исправлен реальный баг (21.06.2026): в таблице были id с дефисом (`claude-sonnet-4-6`, `claude-haiku-4-5`) — у OpenRouter они через точку (`claude-sonnet-4.6`, `claude-haiku-4.5`). С дефисом запрос вернул бы 404/400. Проверено по живому каталогу `GET /api/v1/models`.

### 4.2 Pareto Code Router — код по минимальной цене за нужное качество

Модель `openrouter/pareto-code` сама выбирает САМУЮ ДЕШЁВУЮ модель, проходящую порог качества кода (рейтинг Artificial Analysis). Порог задан в `config.yaml`:
```yaml
openrouter:
  min_coding_score: 0.65   # 0.0–1.0, выше = сильнее/дороже кодер
```
Работает **только** когда вызываемая модель — именно `openrouter/pareto-code` (на любой другой модели поле игнорируется, подтверждено в исходнике `plugins/model-providers/openrouter/__init__.py`). Использовать для задач кодогенерации, где не критично «какая именно модель», важен баланс цена/качество.

### 4.3 Кэш ответов OpenRouter (response cache)

`openrouter.response_cache: true` + `response_cache_ttl: 300` (в `config.yaml`) — Hermes шлёт заголовки `X-OpenRouter-Cache: true` и `X-OpenRouter-Cache-TTL: 300` на каждый запрос к OpenRouter. Если в течение 300 сек повторяется ИДЕНТИЧНЫЙ запрос (та же модель + сообщения + параметры) — OpenRouter возвращает закэшированный ответ **бесплатно**, без повторного биллинга. Не путать с Anthropic prompt caching — это отдельный, edge-уровневый кэш самого OpenRouter.

### 4.4 Резервный провайдер (fallback) — пока НЕ активирован

В `config.yaml` есть `fallback_providers: []` (пусто) — Hermes поддерживает цепочку резервных провайдеров на случай отказа основного (рейт-лимит, 5xx, протухший ключ DeepSeek). OpenRouter — один из ~30 поддерживаемых fallback-провайдеров. Сейчас список пуст — это осознанно НЕ включено в этой версии (поведенческое изменение для прода трогать без явного запроса не стали). Если нужна устойчивость к падению DeepSeek — добавить:
```yaml
fallback_providers:
  - provider: openrouter
    model: deepseek/deepseek-chat   # или любая другая модель-замена
```

> Подключение OpenRouter подтверждено эмпирически (21.06.2026): `GET /api/v1/auth/key` → ключ валиден (лимит $10, остаток $9.99985); реальный запрос `chat_completion` на `anthropic/claude-haiku-4.5` → HTTP 200, ответ "OK", `usage.cost = $0.000034`.

## 5. Роутинг инструментов поиска и исследования

У Hermes несколько источников информации. Выбирай инструмент по типу задачи, а не по привычке (с учётом правила 2 — приоритет MCP):

| Задача | Инструмент |
|--------|-----------|
| Свежий факт / ресёрч из веба + нужны источники-пруфы | **Perplexity** MCP (`sonar` / `sonar-pro`) |
| Глубокое исследование темы, развёрнутый отчёт | **Perplexity** `sonar-deep-research` |
| Анализ / рассуждение поверх найденного в вебе | **Perplexity** `sonar-reasoning-pro` |
| Прочитать конкретный известный URL → чистый markdown | **Firecrawl** `scrape` |
| Узнать карту/структуру сайта (список URL) | **Firecrawl** `map` |
| Лёгкий веб-поиск без синтеза (просто ссылки+сниппеты) | **Firecrawl** `search` |
| Структурированные данные с страницы по схеме (цены, контакты, фичи конкурента) | **Firecrawl** `v2/scrape` с форматом `json` |
| Многостраничный обход сайта (весь блог, каталог) | **Firecrawl** `crawl` |
| Массовый структурированный скрап сайтов (акторы под конкретные площадки) | Apify |
| Документация библиотек / фреймворков | Context7 |
| Рассуждение над уже данным контекстом, большой контекст | Gemini / OpenRouter (см. правило 4) |
| Пульс соцсетей / X (Twitter) | x_search (grok) |

**Ниша Perplexity:** «задал вопрос → синтезированный ответ из живого веба **с источниками**». Это НЕ скрапер конкретного URL и НЕ чтение документации. Бери, когда нужен ответ + доказательства (цитаты).

Модели по стоимости (от дешёвой к дорогой): `sonar` → `sonar-pro` → `sonar-reasoning-pro` → `sonar-deep-research`. Выбирай минимально достаточную.

> Подключение Perplexity по MCP подтверждено эмпирически (21.06.2026): `hermes mcp list` → `perplexity ✓ enabled`, прямой запрос к `api.perplexity.ai` → HTTP 200 с цитатами.

**Ниша Firecrawl:** «есть конкретный URL/сайт → нужен его контент или структура» (НЕ вопрос в свободной форме — за этим к Perplexity). Главный инструмент для **разведки конкурентов и сайтов** (берёт реальную страницу, а не пересказ о ней).

⚠️ `/v1/extract` — **deprecated**, не использовать. Структурированные данные брать через `v2/scrape` с `formats: [{type: "json", schema/prompt}]` — даёт тот же результат заметно дешевле (см. §9, эмпирически: 5 кредитов против 22 у старого `/v1/extract`).

> Подключение Firecrawl подтверждено эмпирически (21.06.2026): прямой запрос `api.firecrawl.dev/v1/scrape` → HTTP 200, `success: true`; проверены `scrape`/`map`/`search`/`v2/scrape json` — все работают на реальном ключе.

## 6. Поддержание активности сервисов (Keep-Alive)

Некоторые подключённые сервисы засыпают при отсутствии активности. Раз в 5 дней (или при любом удобном случае) выполняй профилактический пинг:

| Сервис | Действие | Почему |
|--------|----------|--------|
| Supabase | `list_tables` или `execute_sql SELECT 1` | Засыпает через 7 дней (бесплатный план) |
| Qdrant | `qdrant-find` с любым запросом в коллекцию "foresight" | Засыпает через ~14 дней |
| Pinecone | `list-indexes` | Профилактика |
| n8n | `search_workflows` | Воркфлоу могут деактивироваться |

**Как выполнять:**
- Автоматически — через cron каждые 5 дней (настроен отдельно)
- Вручную — если пользователь жалуется что сервис недоступен
- При старте нового разговора — можно быстро проверить статус Supabase

При обнаружении проблемы (INACTIVE, ошибка соединения) — немедленно попытаться восстановить и уведомить пользователя в Telegram.

## 8. Прозрачность и учёт расходов (ОБЯЗАТЕЛЬНО)

Пользователь хочет понимать, **сколько средств тратится на Hermes**. Поэтому при КАЖДОМ запросе, где использован платный инструмент или не-дефолтная LLM-модель (Perplexity, OpenRouter-модели, Replicate, ElevenLabs, Runway, Firecrawl, Apify, Gemini, Helicone-роутинг и т.п.):

**1. Подписывать инструменты и модели.** В конце ответа давать строку-подпись: какие инструмент(ы) и какие LLM-модели реально работали в этом запросе.

**2. Показывать стоимость запроса.** Брать **фактическую** стоимость:
- Если инструмент возвращает цену в ответе (например Perplexity → `usage.cost.total_cost`) — брать её.
- Если нет — оценивать по таблице `cost-pricing.md` (ставки за токен/символ/секунду/страницу) и помечать как «≈ оценка».

**3. Вести общий счёт.** Каждую платную операцию дописывать в журнал `cost-ledger.md` (дата, инструмент, модель, стоимость, бегущий итог) и показывать обновлённый **Σ всего на Hermes**.

**Формат подписи в конце ответа:**
```
— — —
🔧 Инструменты: Perplexity MCP · модель sonar
💸 Запрос: $0.0050  |  Σ всего на Hermes: $0.0050
```
Если платных инструментов в запросе не было — подпись не нужна (бесплатный ответ дефолтной модели DeepSeek v4 не считаем; при желании можно указать `🔧 DeepSeek v4 · $0.00`).

**Файлы:**
- Журнал: `C:\Users\Unknown\AppData\Local\hermes\cost-ledger.md`
- Ставки: `C:\Users\Unknown\AppData\Local\hermes\cost-pricing.md`

**Источники истины по стоимости:**
- Perplexity — точная цена в каждом ответе API.
- LLM-токены (DeepSeek/OpenRouter) — авторитетно через **Helicone MCP** (уже подключён) — там реальные дашборды расходов.
- Остальные сервисы — оценка по `cost-pricing.md`, уточнять по факту в их биллинге.

## 9. Экономия ресурсов без потери качества

**Главный принцип:** добиваться **максимального качества результата минимальной ценой**.
Где можно сделать запрос дешевле **без ущерба качеству** — делать дешевле.
Где дешёвый путь **не даёт** нужного качества — не экономить: брать более мощный/дорогой инструмент ради результата.

**Приоритет при конфликте:** **качество результата важнее экономии.** Если выбор «дёшево, но хуже» против «дороже, но нужное качество» — выбираем качество (и фиксируем трату по §8).

**Тактики экономии (применять, пока не страдает качество):**
1. **Минимально достаточный уровень модели/инструмента.** Поднимайся по лестнице только при необходимости: `sonar` → `sonar-pro` → `sonar-reasoning-pro` → `sonar-deep-research`; дешёвая LLM (haiku/дефолт) для простого, мощная (opus/reasoning) — для сложного.
2. **Правильный инструмент под задачу** (см. §5). Не гнать дорогой поиск, если нужен конкретный URL (Firecrawl/WebFetch). Не дёргать платную LLM, если ответ уже есть в памяти (Qdrant/Chroma) или в кэше. Внутри самого Firecrawl — `v2/scrape` с `json`-форматом вместо deprecated `/v1/extract` (та же структурированная выжимка, в ~4 раза дешевле по кредитам).
3. **Кэш и память.** Использовать prompt-caching, OpenRouter response-cache (заголовки `X-OpenRouter-Cache`, TTL 300с — см. §4.3, повтор идентичного запроса бесплатен), векторную память — не платить дважды за одно и то же.
4. **Без лишних вызовов.** Сначала подумать — потом звать платный инструмент. Объединять запросы, не делать избыточных/повторных.
5. **Free/локальное при равном качестве.** Локальный TTS/STT вместо платного, если качества достаточно.

§8 (прозрачность) + §9 (экономия) работают в паре: §9 выбирает самый дешёвый достаточный путь, §8 фиксирует и показывает трату.
