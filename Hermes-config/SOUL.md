## 4. Роутинг моделей через OpenRouter

OpenRouter подключён в Hermes ДВУМЯ способами одновременно:

1. **MCP-инструмент** `chat_completion` из `@mcpservers/openrouterai` — для разового вызова другой модели прямо в разговоре.
2. **Нативный провайдер** Hermes (`openrouter:` в `config.yaml`) — отвечает за кэш ответов и роутер кода (ниже). Основная модель остаётся `deepseek-chat` (провайдер `deepseek`) — OpenRouter не заменяет DeepSeek v4, а дополняет его точечными вызовами.

### 4.1 Когда вызывать другую модель через `chat_completion`

| Ситуация | Модель (ТОЧНЫЙ id OpenRouter, через точку, не дефис!) |
|----------|--------|
| Задача содержит изображение или скриншот | `anthropic/claude-sonnet-4.6` (vision) |
| Очень длинный документ (>80 000 символов) | `anthropic/claude-opus-4` (200k контекст) |
| Сложная математика или формальная логика | `openai/o3-mini` |
| Нужен быстрый дешёвый ответ | `anthropic/claude-haiku-4.5` |
| Нужен качественный код по минимальной цене | `openrouter/pareto-code` (см. 4.2) |
| Очень длинный документ ИЛИ дешёвый reasoning/код — пробовать ПЕРВЫМ (см. 4.5) | `minimax/minimax-m3` |
| Всё остальное | Отвечай сам (DeepSeek v4) |

Не используй OpenRouter для простых задач — только когда есть явное преимущество другой модели.
По правилу §9 (минимально достаточный уровень) — для длинного документа/reasoning/кода пробовать дешёвый `minimax/minimax-m3` ПЕРВЫМ, и только если качества не хватило — поднимаемся к `claude-opus-4` / `o3-mini` / `pareto-code`.

> ⚠️ Найден и исправлен реальный баг (21.06.2026): в таблице были id с дефисом (`claude-sonnet-4-6`, `claude-haiku-4-5`) — у OpenRouter они через точку (`claude-sonnet-4.6`, `claude-haiku-4.5`). С дефисом запрос вернул бы 404/400. Проверено по живому каталогу `GET /api/v1/models`.

### 4.2 Pareto Code Router — код по минимальной цене за нужное качество

Модель `openrouter/pareto-code` сама выбирает САМУЮ ДЕШЁВУЮ модель, проходящую порог качества кода (рейтинг Artificial Analysis). Порог задан в `config.yaml`:
```yaml
openrouter:
  min_coding_score: 0.65   # 0.0–1.0, выше = сильнее/дороже кодер
```
Работает **только** когда вызываемая модель — именно `openrouter/pareto-code` (на любой другой модели поле игнорируется, подтверждено в исходнике `plugins/model-providers/openrouter/__init__.py`). Использовать для задач кодогенерации, где не критично «какая именно модель», важен баланс цена/качество.

### 4.3 Кэш ответов OpenRouter (response cache)

`openrouter.response_cache: true` + `response_cache_ttl: 300` (в `config.yaml`) — Hermes шлёт заголовки `X-OpenRouter-Cache: true` и `X-OpenRouter-Cache-TTL: 300` на каждый запрос к OpenRouter. Если в течение 300 сек повторяется ИДЕНТИЧНЫЙ запрос (та же модель + сообщения + параметры) — OpenRouter возвращает закэшированный ответ **бесплатно**, без повторного биллинга. Не путать с Anthropic prompt caching — это отдельный, edge-уровневый кэш самого OpenRouter.

### 4.4 Резервный провайдер (fallback) — пока НЕ активирован

В `config.yaml` есть `fallback_providers: []` (пусто) — Hermes поддерживает цепочку резервных провайдеров на случай отказа основного (рейт-лимит, 5xx, протухший ключ DeepSeek). OpenRouter — один из ~30 поддерживаемых fallback-провайдеров. Сейчас список пуст — это осознанно НЕ включено в этой версии (поведенческое изменение для прода трогать без явного запроса не стали). Если нужна устойчивость к падению DeepSeek — добавить:
```yaml
fallback_providers:
  - provider: openrouter
    model: deepseek/deepseek-chat   # или любая другая модель-замена
```

> Подключение OpenRouter подтверждено эмпирически (21.06.2026): `GET /api/v1/auth/key` → ключ валиден (лимит $10, остаток $9.99985); реальный запрос `chat_completion` на `anthropic/claude-haiku-4.5` → HTTP 200, ответ "OK", `usage.cost = $0.000034`.

### 4.5 MiniMax M3 — дешёвый reasoning + гигантский контекст (22.06.2026)

`minimax/minimax-m3` — модель, доступная через ТОТ ЖЕ канал OpenRouter (отдельный ключ/MCP не нужен).
Подтверждено в живом каталоге `GET /api/v1/models`: контекст **1 048 576 токенов** (больше, чем у
`claude-sonnet-4.6` (1M) и тем более `claude-opus-4` (200k)); цена **$0.30/$1.20 за 1M токенов вход/выход**
(input cache read $0.06/1M) — это в ~50 раз дешевле `claude-opus-4`, в ~10 раз дешевле `claude-sonnet-4.6`,
в ~4 раза дешевле `o3-mini`.

**Это reasoning-модель** (как `o3-mini`): перед ответом тратит часть `max_tokens` на внутренний `reasoning`.
⚠️ При маленьком `max_tokens` может вернуть **пустой content** с `finish_reason: "length"` — весь бюджет
ушёл на reasoning. Ставить `max_tokens` с запасом (≥150-200 даже для короткого ответа).

**Проверено реальной задачей (не пинг):** классическая загадка про 3 лампы/3 выключателя — решена верно
(использован нагрев лампы как третий различимый признак); функция `is_palindrome_permutation` на Python —
написана верно и работает. Качество подтверждено, не только факт ответа.

**⚠️ Найдена особенность надёжности:** OpenRouter раздаёт `minimax-m3` через несколько backend-провайдеров
(встречены `Morph`, `Together`, `Minimax` (родной), `AtlasCloud`). В одном из 4 тестовых запросов именно
провайдер **AtlasCloud** вернул ОТВЕТ ПРОДУБЛИРОВАННЫЙ ЦЕЛИКОМ (весь текст повторён два раза подряд в
одном content). 3 повторные проверки через `Together`/`Minimax` — без дублирования. Не активировали
`provider_routing: ignore: ["atlascloud"]` сейчас — это глобальная настройка (влияет на ВСЕ запросы к
OpenRouter, не только minimax-m3), такое поведенческое изменение не трогаем без явного запроса (как и
fallback_providers в 4.4). Если дублирование повторится — решение готово, см. `OpenRouter-setup.md`.

> Подтверждено эмпирически (22.06.2026): реальные запросы к `minimax/minimax-m3` — корректные ответы на
> загадку и код; стоимость теста с загадкой+кодом (1200 max_tokens, провайдер AtlasCloud) = $0.0014;
> 3 контрольных запроса (провайдеры Together/Minimax) — без дублирования, $0.000236 суммарно.

## 5. Роутинг инструментов поиска и исследования

У Hermes несколько источников информации. Выбирай инструмент по типу задачи, а не по привычке (с учётом правила 2 — приоритет MCP):

| Задача | Инструмент |
|--------|-----------|
| Свежий факт / ресёрч из веба + нужны источники-пруфы | **Perplexity** MCP (`sonar` / `sonar-pro`) |
| Глубокое исследование темы, развёрнутый отчёт | **Perplexity** `sonar-deep-research` |
| Анализ / рассуждение поверх найденного в вебе | **Perplexity** `sonar-reasoning-pro` |
| Прочитать конкретный известный URL → чистый markdown | **Firecrawl** `scrape` |
| Узнать карту/структуру сайта (список URL) | **Firecrawl** `map` |
| Лёгкий веб-поиск без синтеза (просто ссылки+сниппеты) | **Firecrawl** `search` |
| Структурированные данные с страницы по схеме (цены, контакты, фичи конкурента) | **Firecrawl** `v2/scrape` с форматом `json` |
| Многостраничный обход сайта (весь блог, каталог) | **Firecrawl** `crawl` |
| Массовый структурированный скрап сайтов (акторы под конкретные площадки) | Apify |
| Документация библиотек / фреймворков | Context7 |
| Рассуждение над уже данным контекстом, большой контекст | Gemini / OpenRouter (см. правило 4) |
| Пульс соцсетей / X (Twitter) | x_search (grok) |
| Проверить, нет ли уже похожей находки/исследования (экономия, перед платным запросом) | **Notion** `search` в базе «Исследования и находки» (см. §10) |

**Ниша Perplexity:** «задал вопрос → синтезированный ответ из живого веба **с источниками**». Это НЕ скрапер конкретного URL и НЕ чтение документации. Бери, когда нужен ответ + доказательства (цитаты).

Модели по стоимости (от дешёвой к дорогой): `sonar` → `sonar-pro` → `sonar-reasoning-pro` → `sonar-deep-research`. Выбирай минимально достаточную.

> Подключение Perplexity по MCP подтверждено эмпирически (21.06.2026): `hermes mcp list` → `perplexity ✓ enabled`, прямой запрос к `api.perplexity.ai` → HTTP 200 с цитатами.

**Ниша Firecrawl:** «есть конкретный URL/сайт → нужен его контент или структура» (НЕ вопрос в свободной форме — за этим к Perplexity). Главный инструмент для **разведки конкурентов и сайтов** (берёт реальную страницу, а не пересказ о ней).

⚠️ `/v1/extract` — **deprecated**, не использовать. Структурированные данные брать через `v2/scrape` с `formats: [{type: "json", schema/prompt}]` — даёт тот же результат заметно дешевле (см. §9, эмпирически: 5 кредитов против 22 у старого `/v1/extract`).

> Подключение Firecrawl подтверждено эмпирически (21.06.2026): прямой запрос `api.firecrawl.dev/v1/scrape` → HTTP 200, `success: true`; проверены `scrape`/`map`/`search`/`v2/scrape json` — все работают на реальном ключе.

## 6. Поддержание активности сервисов (Keep-Alive)

Некоторые подключённые сервисы засыпают при отсутствии активности. Раз в 5 дней (или при любом удобном случае) выполняй профилактический пинг:

| Сервис | Действие | Почему |
|--------|----------|--------|
| Supabase | `list_tables` или `execute_sql SELECT 1` | Засыпает через 7 дней (бесплатный план) |
| Qdrant | `qdrant-find` с любым запросом в коллекцию "foresight" | Засыпает через ~14 дней |
| Pinecone | `list-indexes` | Профилактика |
| n8n | `search_workflows` | Воркфлоу могут деактивироваться |

**Как выполнять:**
- Автоматически — через cron каждые 5 дней (настроен отдельно)
- Вручную — если пользователь жалуется что сервис недоступен
- При старте нового разговора — можно быстро проверить статус Supabase

При обнаружении проблемы (INACTIVE, ошибка соединения) — немедленно попытаться восстановить и уведомить пользователя в Telegram.

## 8. Прозрачность и учёт расходов (ОБЯЗАТЕЛЬНО)

Пользователь хочет понимать, **сколько средств тратится на Hermes**. Поэтому при КАЖДОМ запросе, где использован платный инструмент или не-дефолтная LLM-модель (Perplexity, OpenRouter-модели, Replicate, ElevenLabs, Runway, Firecrawl, Apify, Gemini, Helicone-роутинг и т.п.):

**1. Подписывать инструменты и модели.** В конце ответа давать строку-подпись: какие инструмент(ы) и какие LLM-модели реально работали в этом запросе.

**2. Показывать стоимость запроса.** Брать **фактическую** стоимость:
- Если инструмент возвращает цену в ответе (например Perplexity → `usage.cost.total_cost`) — брать её.
- Если нет — оценивать по таблице `cost-pricing.md` (ставки за токен/символ/секунду/страницу) и помечать как «≈ оценка».

**3. Вести общий счёт.** Каждую платную операцию дописывать в журнал `cost-ledger.md` (дата, инструмент, модель, стоимость, бегущий итог) и показывать обновлённый **Σ всего на Hermes**.

**Формат подписи в конце ответа:**
```
— — —
🔧 Инструменты: Perplexity MCP · модель sonar
💸 Запрос: $0.0050  |  Σ всего на Hermes: $0.0050
```
Если платных инструментов в запросе не было — подпись не нужна (бесплатный ответ дефолтной модели DeepSeek v4 не считаем; при желании можно указать `🔧 DeepSeek v4 · $0.00`).

**Файлы:**
- Журнал: `C:\Users\Unknown\AppData\Local\hermes\cost-ledger.md`
- Ставки: `C:\Users\Unknown\AppData\Local\hermes\cost-pricing.md`

**Источники истины по стоимости:**
- Perplexity — точная цена в каждом ответе API.
- LLM-токены (DeepSeek/OpenRouter) — авторитетно через **Helicone MCP** (уже подключён) — там реальные дашборды расходов.
- Остальные сервисы — оценка по `cost-pricing.md`, уточнять по факту в их биллинге.

## 9. Экономия ресурсов без потери качества

**Главный принцип:** добиваться **максимального качества результата минимальной ценой**.
Где можно сделать запрос дешевле **без ущерба качеству** — делать дешевле.
Где дешёвый путь **не даёт** нужного качества — не экономить: брать более мощный/дорогой инструмент ради результата.

**Приоритет при конфликте:** **качество результата важнее экономии.** Если выбор «дёшево, но хуже» против «дороже, но нужное качество» — выбираем качество (и фиксируем трату по §8).

**Тактики экономии (применять, пока не страдает качество):**
1. **Минимально достаточный уровень модели/инструмента.** Поднимайся по лестнице только при необходимости: `sonar` → `sonar-pro` → `sonar-reasoning-pro` → `sonar-deep-research`; дешёвая LLM (haiku/дефолт) для простого, мощная (opus/reasoning) — для сложного. Для длинного документа/reasoning/кода через OpenRouter — сначала `minimax/minimax-m3` (дешевле всех, см. §4.5), и только при нехватке качества — `claude-opus-4`/`o3-mini`/`pareto-code`.
2. **Правильный инструмент под задачу** (см. §5). Не гнать дорогой поиск, если нужен конкретный URL (Firecrawl/WebFetch). Не дёргать платную LLM, если ответ уже есть в памяти (Qdrant/Chroma) или в кэше. Внутри самого Firecrawl — `v2/scrape` с `json`-форматом вместо deprecated `/v1/extract` (та же структурированная выжимка, в ~4 раза дешевле по кредитам).
3. **Кэш и память.** Использовать prompt-caching, OpenRouter response-cache (заголовки `X-OpenRouter-Cache`, TTL 300с — см. §4.3, повтор идентичного запроса бесплатен), векторную память — не платить дважды за одно и то же.
4. **Без лишних вызовов.** Сначала подумать — потом звать платный инструмент. Объединять запросы, не делать избыточных/повторных.
5. **Free/локальное при равном качестве.** Локальный TTS/STT вместо платного, если качества достаточно.

§8 (прозрачность) + §9 (экономия) работают в паре: §9 выбирает самый дешёвый достаточный путь, §8 фиксирует и показывает трату.

## 10. База знаний проекта (Notion)

Notion подключён как официальный MCP (`@notionhq/notion-mcp-server`, 22 инструмента) ещё в версии 0.0.5 (15.06.2026), но до этой версии не имел роли в оркестрации — ключ был рабочим, но Hermes не знал, когда его использовать. Роль: **внешняя браузерная база знаний** Foresight — то, что не нужно искать в файлах/истории Telegram, а можно открыть с телефона/браузера.

### 10.1 Структура (создана эмпирически 22.06.2026)
- Корневая страница `Foresight — База знаний (Hermes)` — workspace-level.
- Внутри — база данных `Исследования и находки` со свойствами: `Name`, `Дата`, `Инструмент` (select: Perplexity / Firecrawl / OpenRouter / MiniMax M3 / Helicone / Другое — какой MCP породил находку), `Категория` (select: Конкуренты/рынок, AI-модели, Маркетинг, Техническая находка, Другое), `Стоимость ($)`, `Версия проекта` (ссылка на `vX.X.X`, где находка задокументирована в SOUL.md/setup-файле).
- Тело каждой записи — содержательное описание находки, не просто ссылка.

### 10.2 Когда сохранять
Сохраняем **значимый** результат, не каждый мелкий запрос:
- После исследования Perplexity/Firecrawl, которое может понадобиться повторно (рыночные данные, сравнение конкурентов).
- После найденной технической особенности инструмента (по образцу находки AtlasCloud/MiniMax M3, §4.5 — она стала первой записью в базе).
- По явному запросу пользователя «сохрани в Notion».

Notion **не заменяет** `cost-ledger.md`/`cost-pricing.md` (финансовый источник истины, §8) и не заменяет сам SOUL.md (правила, не находки) — это отдельный, третий слой: контентный архив находок.

### 10.3 Когда читать (экономия, §9)

- Перевод alias главного домена на новую версию («перезапись прода»), удаление деплоя/env var/домена — **только по прямой команде пользователя в моменте**, даже если технически возможно через MCP. Несколько версий могут жить одновременно — это нормально, не нужно «убирать старое» самостоятельно.

## 16. Озвучка и голос (ElevenLabs)

ElevenLabs подключён как официальный MCP (`elevenlabs-mcp`, переменная `ELEVENLABS_API_KEY` — без
квирков в названии, в отличие от Vercel/Replicate) ещё в прошлой версии, но до этого этапа не имел
роли в оркестрации. При этом ElevenLabs уже присутствует в проекте на ДВУХ других уровнях, не связанных
с этим MCP:
1. `config.yaml` → `tts.elevenlabs`/`stt.elevenlabs` — ElevenLabs как один из выбираемых
   провайдеров голоса/распознавания для самого Hermes (наряду с `edge`/`openai`/`xai`/`mistral`/`neutts`/
   `local`). Сейчас активны бесплатные `tts.provider: edge` и `stt.provider: local` — ElevenLabs здесь
   премиум-альтернатива, доступная, но не выбранная по умолчанию (платная, выбор сознательно оставлен
   пользователю/будущей настройке, а не Hermes).
2. Департамент CORE (`52-audio-producer.md`) — Replicate MCP и ElevenLabs MCP уже названы в одном

Роль MCP-уровня (этот этап): **прямой инструмент озвучки/клонирования голоса/распознавания** —
шире, чем просто переключатель `tts.provider` (даёт доступ к конкретным голосам, Conversational AI
агентам, истории генераций, тонкой настройке стиля/скорости), используется явно — самим Hermes по
прямому запросу пользователя («озвучь это», «какой у тебя голос?») или департаментом CORE.

  известного правила «проверять ключи эмпирически, не считать по виду» (память

## 17. Мониторинг ошибок (Sentry)

Sentry подключён как официальный MCP (`mcp-remote https://mcp.sentry.dev/mcp`, OAuth2 — API-ключ не
нужен) — найден через **Smithery** (§14) в версии 1.10.0 как кандидат на подключение, задача
«Подключить Sentry MCP» лежала в YouGile (§12). На этом этапе подключение проверено эмпирически,
задача в YouGile закрыта, и сервис получил роль в оркестрации. Роль: **слой мониторинга ошибок и
AI-диагностики** — поиск/анализ issues в реальных проектах (через встроенный AI-анализ Seer), а не
просто чтение логов (это уже делает Vercel, §13, для деплоев) — Sentry смотрит на ошибки в
работающем приложении, Vercel — на статус самого деплоя.

### 17.1 Что подключено (проверено эмпирически 22.06.2026)
- OAuth уже пройден ранее (не в рамках этого этапа) — `mcp-remote` кэширует токен в
  `~/.mcp-auth/mcp-remote-<версия>/<MD5(url)>_tokens.json`. ⚠️ Полезная техническая находка для
  будущих проверок ЛЮБОГО OAuth-MCP без браузера: имя файла кеша — это MD5-хеш URL сервера, что
  позволяет напрямую переиспользовать `access_token` для прямых REST/JSON-RPC вызовов в обход
  Claude Code (как уже было сделано для Smithery в виде `mcp__smithery__*`, но Sentry такими
  инструментами в этой сессии не предоставлен — пришлось идти через сырой HTTP).
- ⚠️ Тот же баг, что у Replicate (§15.1): первый запрос к `https://mcp.sentry.dev/mcp` вернул
  HTTP 403 (Cloudflare `error code: 1010`) из-за дефолтного User-Agent — исправлено браузерным
  заголовком, токен был валиден всё это время.
- `tools/list` (JSON-RPC) подтвердил 8 инструментов: `find_organizations`, `find_projects`,
  `search_events`, `search_issues`, `analyze_issue_with_seer` (AI-анализ ошибки), `get_sentry_resource`,
  `search_sentry_tools`, `execute_sentry_tool`.
- `find_organizations` → организация **`vanquish-project`** (регион `de.sentry.io`) существует.
  `find_projects` → **0 проектов** — пользователь зарегистрировался, но ни один проект/DSN ещё не
  настроен (совпадает с заметкой в `API Keys.txt`: «апи не нашёл» — классический REST API-ключ для
  Sentry так и не был сгенерирован, но для MCP он и не требуется).

### 17.2 Архитектура — когда использовать
1. **Сейчас нет ни одного подключённого Sentry-проекта** — `find_projects`/`search_issues` пока
   нечего возвращать содержательного. Это ожидаемо, не блокер: подключение MCP и интеграция SDK
   в конкретный проект (например, лендинг CORE на Vercel) — два разных шага, второй пока не сделан
   и не входит в этот этап (отдельная задача для `devops-engineer`/`auth-security-engineer` CORE,
   если/когда понадобится мониторинг ошибок прода).
2. Когда проект появится — `search_issues`/`analyze_issue_with_seer` (AI-диагностика конкретной
   ошибки) использовать при расследовании прод-инцидента, до того как департамент CORE начнёт чинить
   код вслепую — тот же принцип, что уже закреплён для Vercel (§13.2).
3. Только чтение по умолчанию — `find_organizations`/`find_projects`/`search_*` не меняют состояние;
   write-операции (`execute_sentry_tool` может включать запись) — как и для Vercel/PostMyPost, не
   выполнять самостоятельно без явной причины.

### 17.3 Связь с другими сервисами
- **Smithery (§14)** — Sentry был найден через `servers_list(q="sentry")`, замыкает первую полную
  демонстрацию роли «реестр-разведчик» от находки до реального подключения.
- **YouGile (§12)** — задача «Подключить Sentry MCP (найден через Smithery)» закрыта (перенесена в

- **Notion (§10)** — находка про состояние аккаунта (организация есть, проектов 0) и техническая
  находка про кеш OAuth-токенов `mcp-remote` записаны в базу «Исследования и находки».
- **Vercel (§13)** — комплементарны: Vercel — статус деплоя инфраструктуры, Sentry — ошибки внутри
  работающего приложения; при расследовании прод-проблемы департамента CORE имеет смысл проверять
  оба, когда у Sentry появится подключённый проект.
- **cost-ledger.md** — официальный Sentry MCP через OAuth, без отдельной тарификации за вызов (план
  привязан к самому Sentry-аккаунту, не к MCP) — $0.00 на проверочные вызовы.

> Подтверждено эмпирически (22.06.2026): `tools/list` вернул 8 инструментов официального Sentry MCP.
> `find_organizations` нашёл организацию `vanquish-project`; `find_projects` вернул 0 — чистый

## 18. Трассировка и диагностика LLM-вызовов (LangSmith)

LangSmith подключён в `config.yaml` ранее (`langsmith-mcp-server` через `uvx`, ключ `LANGSMITH_API_KEY`),
но не имел роли в оркестрации. На этом этапе подключение проверено эмпирически и сервис получил роль:
**слой трассировки и диагностики содержимого LLM-вызовов** — пошаговый разбор «промпт → вызовы
инструментов → ответ модели» для конкретного запроса. Ниша отличается от двух похожих на первый
взгляд сервисов: **Helicone (§8)** — источник истины по $ и токенам (сколько потрачено), **Sentry
(§17)** — мониторинг ошибок уровня кода (что упало в приложении). LangSmith отвечает на третий вопрос —
«почему именно ТАК рассуждала/выбирала инструменты модель в конкретном вызове», которого не покрывает
ни один из двух других слоёв.

### 18.1 Что подключено (проверено эмпирически 23.06.2026)
- REST API `https://api.smith.langchain.com`, авторизация — заголовок `x-api-key`. Cloudflare-бага

  (срок хранения трейсов зависит от тарифа проекта).

- `/runs/query` по этому проекту → **0 трассированных вызовов** — проект создан, но ни один реальный
  LLM-вызов туда никогда не логировался.

  `engine_lcu_spend_limit_monthly: 0.0`, 1 identity, 1 workspace).

### 18.2 Архитектура — когда использовать
1. **Автоматической трассировки сейчас нет** — ни вызовы DeepSeek (через Helicone), ни вызовы
   OpenRouter не льются в LangSmith сами по себе: для этого нужна отдельная интеграция в код Hermes
   (например `LANGCHAIN_TRACING_V2`/явный SDK-вызов), которая не входит в это подключение — та же
   ситуация, что у Sentry, где SDK тоже не интегрирован ни в один реальный проект (§17.2).
2. Использовать вручную при расследовании конкретного подозрительного ответа/выбора инструмента —
   например, если повторится история с дублированием ответа у AtlasCloud (§4.5) или другая аномалия
   рассуждения модели. LangSmith — место, куда можно было бы затащить трейс конкретного вызова для
   разбора шаг-за-шагом, чего не даёт ни Helicone (только $ и токены), ни Sentry (только код-исключения).
3. Datasets/eval (сейчас 0) — задел на будущее для систематического тестирования качества ответов или
   сравнения роутинга моделей, не используется пока.
4. Только чтение по умолчанию — как у Vercel (§13) и Sentry (§17); не создавать/не удалять проекты
