# LangSmith MCP — настройка и архитектура трассировки LLM-вызовов (добавлено 2026-06-23, версия 1.14.0)

LangSmith был подключён в `config.yaml` ранее (`langsmith-mcp-server` через `uvx`), но не имел роли в
оркестрации. На этом этапе подключение проверено эмпирически и сервис получил роль.

## Где настроено (файл в `AppData\Local\hermes\`, в репо не коммитится)
**`config.yaml` → `mcp_servers.langsmith`**:
```yaml
langsmith:
  command: uvx
  args: ["langsmith-mcp-server"]
  env:
    LANGSMITH_API_KEY: lsv2_pt_***
  enabled: true
```
Ключ уже был в конфиге и валиден — проверен прямыми REST-вызовами к `https://api.smith.langchain.com`
(заголовок `x-api-key`), без отдельного MCP-сеанса (та же модель проверки, что использовалась для
Replicate/Sentry — напрямую к API сервиса, в обход Claude Code).

## Архитектура
Полное описание — `SOUL.md` §1. Кратко: слой трассировки и диагностики содержимого LLM-вызовов —
пошаговый разбор «промпт → вызовы инструментов → ответ модели» для конкретного запроса. Ниша отличается
от двух похожих сервисов из удалённой оркестрации (см. `Departments/Инструкция к оркестрации 3
(Неиспользуемая версия).md`): **Helicone** — сколько потрачено ($/токены), **Sentry** — что упало в
коде приложения. LangSmith — почему именно так рассуждала/выбирала инструменты модель.

## Проверка (эмпирически, 23.06.2026)
- `GET /info` → версия `0.16.6`, `instance_flags.trace_tier_duration_days: {longlived: 400, shortlived: 14}`.
- `GET /sessions` (LangSmith называет проекты трассировки «sessions») → один предсуществующий проект
  **«foresight»** (id `42ed25a9-a375-4953-ae26-48dc3fbb521f`, `created_at: 2026-06-08`) — создан
  **раньше начала** этой оркестрации (21.06.2026), не результат текущей работы. `trace_tier: "shortlived"`
  (14 дней хранения трейсов на этом тарифе).
- `POST /runs/query` (фильтр по этому `session`) → `{"runs": [], ...}` — **0 трассированных вызовов**:
  проект создан, но никогда не получал реальных LLM-трейсов.
- `GET /datasets` → `[]` — **0 датасетов** (функциональность офлайн-оценки/eval не использовалась).
- `GET /repos` (хаб промптов) → вернул только публичный community-промпт `hardkothari/prompt-maker`
  (301 likes, 214 912 downloads) — это публичный шаблон LangChain Hub, не собственный контент.
- `GET /orgs/current` → организация **«Personal»**, **`plan_tier: "free"`**,
  `engine_lcu_spend_limit_monthly: 0.0`, `max_identities: 1`, `max_workspaces: 1`.
- Cloudflare-бага (403 при дефолтном User-Agent, см. `Replicate-setup.md`/`Sentry-setup.md`) не
  возникло — проверяли с браузерным User-Agent с первого запроса, защитно, по уже известному паттерну.

## Ключевая находка: подключено, но не интегрировано
В отличие от Sentry (где SDK просто не подключён ни к одному проекту), здесь ситуация симметричная:
**MCP-сервер и ключ работают, но сам Hermes не отправляет туда трейсы своих вызовов** — нужна отдельная
интеграция в код (например `LANGCHAIN_TRACING_V2=true` или явные SDK-вызовы при каждом LLM-запросе),
которая не входит в это подключение. Поэтому роль на этом этапе — задел на будущее (датасеты/eval) +
ручной инструмент точечной диагностики конкретного инцидента, а не автоматический постоянный слой,
как Helicone уже является для затрат.

## Конкретная связь сервисов
- **OpenRouter** — потенциальная точка применения (роль OpenRouter описана только в архиве после
  чистки git-истории 23.06.2026): если потребуется системно разобрать, почему `AtlasCloud`-провайдер
  `minimax/minimax-m3` дублировал ответ, LangSmith — инструмент для трассировки конкретного такого
  вызова шаг-за-шагом, а не разовая ручная проверка.
- **Notion** — находка о состоянии аккаунта (1 пустой проект, бесплатный тариф, 0 трейсов/датасетов)
  фиксируется в базе «Исследования и находки», по аналогии с остальными сервисами.
